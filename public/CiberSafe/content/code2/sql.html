SQL Injection

 

O que é?

A injeção de SQL (SQLi) é uma vulnerabilidade de segurança da Web que permite que um invasor interfira nas consultas que uma aplicação faz na sua base de dados.
Geralmente permite que o invasor tenha acesso a dados sensíveis. Isso pode incluir dados pertencentes a outros utilizadores ou quaisquer outros dados que a própria aplicação possa conter.
Em diversos casos, o invasor pode modificar ou excluir esses dados, causando alterações persistentes no conteúdo ou no comportamento da aplicação.

Qual o impacto de um ataque por “SQL Injection”?

Um ataque de injeção de SQL bem-sucedido pode resultar em acessos não autorizados de dados confidenciais, como passwords, cartões de crédito ou informações pessoais de um determinado utilizador.
Em alguns casos, o invasor pode criar uma “backdoor” nos sistemas de uma organização, levando a um comprometimento de longo prazo que pode passar despercebido por um longo período.

Exemplos de “SQL Injection”

Alguns exemplos de injeção de SQL incluem:
•

Alterando a lógica de uma aplicação

Onde o invasor pode alterar uma consulta para interferir com a lógica da aplicação.
•

Injeção cega de SQL

Onde o resultado de uma consulta não retorna a resposta da aplicação.

Alterando a lógica de uma aplicação

Considere uma aplicação que permite que os utilizadores façam login com nome e password. Se um utilizador enviar o nome de utilizador “joao” e a password ”Teste1234?”, a aplicação vai verificar as credenciais executando a seguinte consulta SQL:

SELECT * FROM users WHERE Username = "joao" AND Password = "Teste1234?"
Se a consulta retornar os detalhes o utilizador, o login foi bem-sucedido. Caso contrário, é rejeitado.
O invasor pode efetuar login como qualquer usuário sem precisar de uma password, simplesmente usando a sequência de comentário SQL. Ou seja, remove a verificação de senha da cláusula WHERE da consulta.

Injeção cega de SQL

Muitas instâncias de injeção de SQL são vulnerabilidades cegas. Isso significa que a aplicação não retorna os resultados da consulta SQL ou os detalhes de erros da base de dados. Vulnerabilidades cegas ainda podem ser exploradas para ter acesso a dados não autorizados, mas as técnicas envolvidas são geralmente mais complicadas e difíceis de se executar.
Dependendo da natureza da vulnerabilidade e da base de dados envolvida, as seguintes técnicas podem ser usadas para explorar vulnerabilidades de injeção cega de SQL:
•
Alterar a lógica de consulta
Alterar a lógica de consulta levando a acionar uma diferença na resposta da aplicação, dependendo da veracidade de uma única condição. Isso pode envolver a introdução de uma nova condição em uma variável booleana ou o acionamento condicional de um erro, como uma divisão por zero.
•
Acionar um atraso no processamento da consulta
Acionar um atraso no processamento da consulta, permitindo inferir a veracidade da condição com base no tempo que a aplicação leva a responder.
•
Possível roubar dados
É possível roubar dados a partir de técnicas do tipo “OAST”, com base no acionar uma interação de rede fora de banda, onde se capta os dados e transfere para o sistema, onde o invasor tem total controlo.

Como detetar um “SQL Injection”?

A maioria das vulnerabilidades de injeção de SQL pode ser encontrada de forma rápida usando um “scanner” de vulnerabilidades da Web, como por exemplo o “Burp Suite”.
A injeção de SQL é detetada manualmente usando um conjunto sistemático de testes em cada ponto de entrada da aplicação. Isso normalmente envolve:
•
Envio de caractere de aspas simples ' e procura de erros ou outras anomalias.
•
Submeter alguma sintaxe específica de SQL avaliadas por um o valor base (original) do ponto de entrada e para um valor diferente e procurar diferenças sistemáticas nas respostas resultantes da aplicação.
•
Submeter condições booleanas como OR 1=1 e OR 1=2 e procurar diferenças nas respostas.
•
Envio de cargas projetadas para acionar atrasos de tempo quando executado uma consulta SQL, procurando assim diferenças no tempo da resposta.
•
Envio de cargas OAST projetadas para acionar uma interação de rede fora de banda quando executado uma consulta SQL, monitorizando quaisquer interações resultantes

Como prevenir ataques “SQL Injection”?

Existem várias maneiras eficazes de impedir a ocorrência de ataques SQLI, bem como protegê-los, caso ocorram.
A primeira coisa a fazer é a validação dos inputs, onde vai identificar entradas ilegítimas feitas pelo utilizador.
Embora a validação dos inputs deva sempre ser considerada a melhor prática, raramente é uma solução infalível. A realidade é que, na maioria dos casos, simplesmente não é viável mapear todas as entradas legais e ilegais, pelo menos sem causar interferências na experiência do utilizador e na funcionalidade da aplicação.
Por esse motivo, uma Firewall para aplicações web é instalado para filtrar o SQLI, bem como outras ameaças online. Para fazer isso, a Firewall normalmente depende de uma lista grande e constantemente atualizada de comandos SQL usados por esse ataque, permitindo assim identificar e eliminar consultas SQL maliciosas.
